# Solution Strategy: BNPL-трекер

## Problem Statement (SCQA)

- **Situation:** Пользователи Клёво импортируют банковские CSV и видят транзакции
- **Complication:** Платежи по рассрочке (Долями/Сплит/Подели) разбросаны по выписке и не сгруппированы — пользователь не знает, сколько ещё должен платить
- **Question:** Как автоматически обнаружить и агрегировать BNPL-обязательства из уже импортированных транзакций?
- **Answer:** Keyword-детекция провайдера + группировка по временным паттернам + хранение как BnplObligation

## First Principles Analysis

1. BNPL = покупка разбита на N равных платежей с фиксированным интервалом
2. Каждый платёж отображается в выписке как отдельная транзакция
3. Транзакции одного BNPL имеют: одинакового провайдера, одинаковую сумму, регулярный интервал
4. Следовательно: можно детектировать через (provider_keyword, amount, interval) кластеризацию

## Root Cause Analysis (5 Whys)

1. Почему пользователь не знает размер BNPL-долга? → Нет агрегации
2. Почему нет агрегации? → Данные разбросаны по нескольким провайдерам
3. Почему данные разбросаны? → Каждый банк/провайдер отдельное приложение
4. Почему не используют одно приложение? → Нет универсального агрегатора для RU BNPL
5. **Root Cause:** Отсутствие единой точки консолидации BNPL-данных из CSV-выписок

## Recommended Approach

### BnplDetector Service
```
transactions → filter by BNPL keywords → group by (provider, amount, ~interval) 
→ sort by date → compute installmentsPaid, nextPaymentDate → BnplObligation[]
```

### Key Design Decisions

**1. Keyword dictionary approach (not ML)**
- Хардкодим паттерны провайдеров (Долями/Сплит/Подели)
- Простота + высокая точность на известных провайдерах
- Extensible: добавить нового провайдера = 1 строка в словаре

**2. Группировка по (provider + amountBucket + interval)**
- amountBucket: округляем до ближайших 100 коп (учитываем ±1 коп расхождения)
- intervalBucket: 14±2 дней или 30±3 дней
- Если 2+ транзакции совпадают → одно BNPL-обязательство

**3. Проекция nextPaymentDate**
- nextPaymentDate = lastPaymentDate + frequencyDays
- Если nextPaymentDate < now → статус overdue
- Если paidInstallments >= totalInstallments → статус completed

**4. totalInstallments estimation**
- Долями/Подели: всегда 4
- Сплит: определяем по количеству уже совершённых с учётом паттерна (fallback: 6)
- Яндекс Сплит: fallback 6

### Contradictions Resolved (TRIZ)

| Противоречие | Принцип | Решение |
|---|---|---|
| Точность vs простота детекции | #10 Предварительное действие | Предиктивная группировка на этапе сканирования |
| Неизвестное totalInstallments | #25 Самообслуживание | Оценка по паттерну, уточняется при появлении новых платежей |

## Risk Assessment

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Банк меняет формат merchant name | Средняя | Среднее | Fuzzy matching + постоянное пополнение словаря |
| Ложные срабатывания | Низкая | Среднее | Пользователь может скрыть обязательство |
| Нет BNPL в выписке пользователя | Высокая | Низкое | UI: «BNPL-транзакций не найдено» |
